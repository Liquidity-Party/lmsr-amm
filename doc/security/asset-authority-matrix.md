# Asset / Authority Matrix ‚Äî LMSR-AMM

**Status:** First-pass matrix
**Companion docs:** `checklist.md` (what classes to look for).

## Purpose

This is the deliverable that replaces prose review per the process doc. Each cell answers, for one (asset √ó external function) pair:

> _Can this function move this asset, and if so, what authorization gates the movement, and where in the code is that gate enforced?_

**Empty cells are bugs.** A blank where movement is possible means an unenforced authorization path. Cells documented "no movement" must be true by code inspection, not by assumption.

## Conventions

- **Direction in cell:** `IN` = function pulls this asset into the pool / contract; `OUT` = function pushes this asset out; `IN/OUT` = both.
- **Auth gate:** the require/check that authorizes the movement. `file:line` points to the enforcement site.
- **`‚Äî`** = function does not move this asset (verified by code inspection).
- **`(view)`** = function is read-only; cannot move anything.
- **Modifiers** are listed as defense-in-depth. `nonReentrant` blocks cross-fn re-entry. `killable` halts after `kill()`. `native()` refunds residual `msg.value` to `msg.sender` post-body.
- **Permit2 ack:** Permit2 itself enforces signature/nonce/deadline; the pool's only obligation is to bind the witness hash so the signature is intent-specific.

### B.2 Empty-cell audit (B.1)

Cells deliberately left as `‚Äî` and the reason each is structurally impossible (must be re-verified per change):

- `swap √ó LP_p / LPA_p^s` ‚Äî swap path never touches `_balances` / `_allowances`; only `_cachedUintBalances`, `_protocolFeesOwed`, `_lmsr.qInternal` are written. Verify by grep: no `_erc20Mint`/`_erc20Burn`/`_erc20Approve` call in `PartyPool.swap` or `_quoteSwapExactIn`.
- `swapMint / mint / initialMint √ó LPA_p^s` ‚Äî these only mint LP to `receiver`; they do not consume an LP-token allowance because there is no LP-token to debit yet (the mint _creates_ it).
- `burn / burnSwap √ó A_i^p / D_i / S_p` ‚Äî burn paths do not pull reserve tokens; they only push them out and burn LP. No `_receiveTokenFrom*` call in `burn`/`burnSwap`. (Confirmed: only `_sendTokenTo` appears.)
- `flashLoan √ó A_i^p / D_i / S_p / LP_p / LPA_p^s` ‚Äî flash repayment uses the _receiver_'s allowance (`Extra.sol:102`), not a swapper's. The receiver explicitly approves the pool inside its `onFlashLoan`. The cell `(A_receiver_i √ó flashLoan)` is captured by the `R_i` row's "round-trip" entry.
- `collectProtocolFees √ó A_i^p / D_i / S_p / LP / LPA / E` ‚Äî protocol-fee collect only reads `_protocolFeesOwed`, transfers from pool, updates `_cachedUintBalances`. No external auth pull.
- `setProtocolFeeAddress / kill √ó everything except governance state` ‚Äî admin operations, no token movement. Storage write only.
- `receive() √ó everything except E` ‚Äî the receive function is a guard, not a movement function; it accepts ETH only from `WRAPPER`.

### C.3 Outstanding `newPool` consideration

The 3-overload `newPool` accepts a `payer` for initial deposits; when `payer != msg.sender` the `safeTransferFrom` consumes `payer`'s allowance to the _pool_ (not the planner). Because `onlyOwner` gates `newPool`, the only entity that can do this is the planner owner. The risk surface is therefore _trust in the planner owner_, not arbitrary external callers ‚Äî but **third-party allowances to pools at deterministic CREATE2 addresses are theoretically pre-grantable**. Listed as item ¬ßH.2 below.

## D. Matrix ‚Äî `PartyConcierge`

The Concierge is a router that translates user-facing ERC20 token addresses into pool-index calls and uses the pool's _callback funding_ mode (so users approve the **Concierge**, not each pool). Auth model: a transient-storage `(_cbUser, _cbPool, _cbEthBudget, _cbMode)` record is set by the entry function, validated by the callback, and cleared afterward. `_cbPool != 0` doubles as the reentrancy guard; `_cbEthBudget` snapshots `msg.value` so the callback's auto-wrap branch cannot consume pre-stuck ETH; `_cbMode` selects the funding path (APPROVAL vs PERMIT2). The Concierge offers two parallel surfaces: APPROVAL/native (`swap`, `mint`, `burn`, `swapMint`, `burnSwap`) and Permit2 (`swapPermit2`, `swapMintPermit2`); both share the same transient-context auth chain.

### D.2 Empty-cell audit (Concierge)

- `liquidityPartySwapCallback √ó LP` ‚Äî callback never touches the LP token. The three callback branches (native auto-wrap `:77-82`, Permit2 pull `:86-101`, allowance `safeTransferFrom` `:104`) all move only the input reserve token `t` into the pool.
- `liquidityPartySwapCallback √ó E` ‚Äî the native auto-wrap branch (`:77-82`) deposits `msg.value`-budgeted ETH into the pool's WRAPPER and forwards the wrapped token to the pool. The Concierge never sends raw ETH to the pool; the pool's `receive()` (which only accepts ETH from WRAPPER) is not invoked from this path.
- Concierge's `burn`/`burnSwap` √ó `A_i^user` (reserve token allowance) ‚Äî burn paths take _LP token_ from the user, not reserve tokens; the basket comes out of the pool. No reserve-allowance consumption.
- Concierge has no admin functions: no owner, no fee, no setter. State is fully transient.

### D.3 Concierge auth-chain integrity check

The transient context (`_cbUser`, `_cbPool`, `_cbEthBudget`, `_cbMode`) is the entire authorization for `liquidityPartySwapCallback`. Three invariants must hold for any change:

1. **Set-before-call, cleared-after.** `_beginCall` must run _before_ any external call; `_endCall` runs after. Any new path that calls a pool must follow this pattern. (Not enforced by a modifier; any new entry needs a manual check.)
2. **No external call between `_beginCall` and `_endCall` other than the pool itself.** A nested call to a non-pool contract during the in-flight window could spoof the callback, since `_cbPool` is a single address. Currently no such call exists in any entry; new functions must preserve this.
3. **`_cbEthBudget` bounds the native auto-wrap branch.** The callback's wrap-from-balance step (`:77-82`) checks `_cbEthBudget >= amount` and decrements; this prevents the callback from silently consuming pre-stuck ETH on `address(this)` (any leftover lands in `sweepEth`'s post-body refund). New entry points that mutate the wrap budget must preserve this snapshot semantics.

### D.4 Open items

- **D.4.1** _Concierge holds LP token in `burn`/`burnSwap`._ The two-step `safeTransferFrom(user‚ÜíConcierge)` then `pool.burn(Concierge as payer)` means Concierge transiently holds the LP token. If the pool burn reverts after the transfer, the LP is stranded in the Concierge; if a subsequent attacker call to a different pool's `burn` could pass `payer=Concierge` and consume that LP, that would be a drain path. ## F. Cross-cutting invariants implied by the matrix

These are restated for the invariant suite (process doc ¬ß3.4). Each is enforceable as a multi-actor Foundry property:

1. **Allowance theft is impossible.** For any token `t`, holder `H`, attacker `A`: `H` granting any allowance to the pool (or Concierge) cannot be consumed except by a tx where `msg.sender == H` (APPROVAL/PREFUNDING), or by a pool callback where `payer == H` (callback funding), or by a Permit2 transfer signed by `H`. **Matrix support:** `A_i^p √ó {swap, swapMint, mint}` cells; Concierge `A_i^user √ó liquidityPartySwapCallback` chain.
2. **Balance reconciliation.** For every `i`, every state-mutating function preserves `balanceOf(pool, t_i) == _cachedUintBalances[i] + _protocolFeesOwed[i]` (modulo unaccounted donations, which are absorbed into `D_i`). Includes the v1 same-token regression test.
3. **No cell drift on refactor.** Any PR that adds an external function or changes a `_receiveToken*` / `_sendToken*` / `_erc20*` site must add or modify rows here. Reviewer's job: confirm the diff in this matrix matches the diff in code.
4. **PREFUNDING payer-gate (not race-proof).** `D_i` is consumable only by a `(payer, msg.sender)` pair where `msg.sender == payer` (`PartyPoolBase.sol:216`), which blocks `payer`-spoof / allowance theft. It does **not** bind `D_i` to its depositor: a `payer == msg.sender` caller can consume a stranger's non-atomic prefund. That front-run is accepted by design ‚Äî PREFUNDING is intended only for integrating contracts that bundle the deposit and call atomically in one tx (see `Funding.PREFUNDING`). Documented donation surplus goes to LPs, not to `msg.sender`.
5. **Permit2 witness intent-binding.** Every Permit2-bound flow must pass a witness that captures all caller-controllable fields that affect asset movement. Already enforced by `SwapWitness` (`PartyPool.sol:212-214`), `MintWitness` (`MintImpl.sol:233-235`), `SwapMintWitness` (`MintImpl.sol:474-476`).

## G. Differential review against canonical AMMs

Process doc ¬ß3.3 asks: for every external function, write the analogous signature in Uniswap V2/V3, Curve, Balancer; document each divergence. The relevant high-level divergences for matrix-impacting decisions are:

- **`payer` parameter.** Uniswap V2/V3 routers use `msg.sender` as the implicit payer. We expose `payer` to support callback-funding (router/aggregator pattern, c.f. Uniswap V3 `IUniswapV3SwapCallback`). The post-fix gate `msg.sender == payer` for APPROVAL/PREFUNDING aligns with Uniswap; the divergence is preserved only in callback mode.
- **Permit2 with witness.** Matches Uniswap UniversalRouter pattern. Witness type strings differ per entry point.
- **Same-token swap rejection.** Uniswap V2/V3 reject by virtue of having one pool per pair; Curve/Balancer reject explicitly. We now reject explicitly at `PartyPool.sol:193`.
- **Anyone-callable `collectProtocolFees`.** Uniswap V3 `collectProtocol` is owner-only. Anyone-callable here is safe because the recipient is fixed (only owner can change `protocolFeeAddress`). Documented divergence; no risk.
- **Killable pool.** Curve/Balancer have admin pause/kill. Uniswap V2/V3 don't. Our `kill()` is one-way and disables only the entry-points marked `K`. Consistent.

The full per-function table ‚Äî every external on `PartyPool`, `PartyPlanner`, and `PartyConcierge` against Uniswap V2, Uniswap V3, Curve StableSwap/CryptoSwap, and Balancer V2 ‚Äî lives in `doc/security/differential-review.md`. That document closes `checklist.md` ¬ßO.8.

## I. How to update this matrix on a PR

Per process doc ¬ß5: any PR touching `_receiveToken*`, `_sendToken*`, `_erc20*`, `safeTransfer*`, `Address.functionCall`, or any `external`/`public` non-view function must update the affected cells, with the file:line in the cell pointing to the _new_ line numbers. Reviewer confirms diff-here matches diff-in-code; an unchanged cell after a code change to its enforcement site is the most common review failure mode.
