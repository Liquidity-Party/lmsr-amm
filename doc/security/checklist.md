# Security Review Checklist — LMSR-AMM

**Status:** First-pass flat checklist, distilled from public catalogs. Drives the per-function asset/authority matrix described in `security-review-process.md` §3.1.
**Sources consulted:**

- `security/references/DefiVulnLabs/` — 48 pattern PoCs (`SunWeb3Sec/DefiVulnLabs`)
- `security/references/building-secure-contracts/` — Trail of Bits dev guidelines + Token Integration + Rekt Test (`crytic/building-secure-contracts`)
- `security/references/DeFiHackLabs/` — incident catalog (grepped for router/arbitrary-call/approval patterns)

**How to use.** This is a flat catalog, one row per known pattern. For each row:

1. Decide **Applicable?** for this codebase (Y / partial / N — N requires a one-line "why not").
2. Fill **Where to look** with `file.sol:line` pointers to the entry points where the pattern _could_ manifest.
3. Mark **Status**: `TODO` (unreviewed), `OK` (reviewed and safe), `MITIGATED` (fixed, with PR/commit ref), `BUG` (open issue).

When all "Applicable" rows are non-`TODO`, the checklist pass is complete. The matrix in §3.1 is built from the `OK`/`MITIGATED` rows that involve token movement or authorisation.

Conventions: `(F)` = highest-severity historical incidents, must verify before any deploy. Unreferenced rows are rooted in DefiVulnLabs unless prefixed `[BSC]` (building-secure-contracts), or `[HACK]` (DeFiHackLabs incident class).
